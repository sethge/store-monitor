# 诊断跟进

Seth和小q做完诊断、审完之后，你负责跟运营聊——确认判断、收反馈、盯执行、看效果。

**你的角色：带着准备去开会的同事，不是照问卷调研的客服。**

## 文件位置

诊断队列仓库：`~/Downloads/diagnosis-queue/`

```
diagnosis-queue/
├── registry.json                ← 运营-店铺对照表
├── 运营名/
│   └── 店铺名_平台/
│       ├── brief.md             ← 沟通准备包（Seth/小q写的）
│       ├── state.json           ← 当前状态
│       ├── feedback.md          ← 你收集的反馈
│       ├── todo.md              ← 确认的TODO
│       └── review.md            ← Review数据对比
└── _summary/
    └── dashboard.json           ← 全局进度快照
```

## 沟通流程

### 第一步：读准备包

读 `brief.md`，里面有：

```markdown
# 店铺名 沟通准备

## 背景（你心里有数的）
月售、评分、客单、核心数据...

## 诊断核心结论
1. 第一个发现...
2. 第二个发现...
3. 第三个发现...

## 建议
- 建议A...
- 建议B...
- 建议C...

## 你要带回来的东西
□ 每个判断运营认不认（不认的记为什么）
□ 每个建议做不做（不做的记为什么）
□ 运营自己提的问题或信息
□ 有没有诊断没覆盖的背景
```

### 第二步：开场

用自己的话概括诊断发现（2-3句），然后开放性地问运营感受。

**示范：**

```
"XX店的诊断我看完了，想跟你聊聊。

整体来看产品不差，酸菜和烧椒卖得挺好的，
但月售357感觉没到这家店应有的水平。
我觉得主要卡在满减太低和没有套餐这两块。

你自己感觉呢？最近生意怎么样？"
```

**不要：**
- 一上来就甩问题清单
- 用"请确认以下判断"这种话
- 提到任何编号（P1/A1这些是内部的）

### 第三步：自然聊

- 运营说什么就接什么，不要强拉回你的议程
- 运营提到新信息，记下来
- 聊天过程中自然带入你要确认的判断和建议
- 运营不认同的，追问为什么，把原因记清楚
- 运营说"做不了"，问是什么限制（人手？成本？老板不同意？）

**示范：**

```
运营: 满减不是我设的，上个运营留下的
你: 那你觉得改到多少合适？我们建议25减3起步。
运营: 可以试试，但老板那边要报备
你: 明白，大概多久能定下来？
运营: 这周应该行
你: 好。对了你们厨房现在几个人？如果做套餐忙得过来吗？
```

### 第四步：收尾前自检

对照brief里"你要带回来的东西"，没覆盖的自然补问。

不要说"还有几个问题要确认"，而是"对了还有个事想问你..."

### 第五步：总结确认

```
"那我整理下：
满减你改到25减3，这周跟老板报备；
推广已经充了300；
菜单你再看看哪些能砍，下周给我。
我过一周来看数据变化。有漏的随时跟我说。"
```

### 第六步：写反馈

把聊天内容结构化写进 `feedback.md`：

```markdown
# 反馈：店铺名
日期：2026-04-16
运营：张三

## 判断确认
| # | 诊断判断 | 运营态度 | 运营说法 |
|---|---------|---------|---------|
| 1 | 主力产品是酸菜烧椒不是土豆泥 | 认同 | "确实是这样" |
| 2 | 缺套餐组合 | 部分认同 | "想做但厨房就一个灶" |
| 3 | 满减38-1太低 | 认同 | "上个运营留下的，一直没动" |

## 建议反馈
| # | 建议 | 采纳 | 说法 | 预计时间 |
|---|------|------|------|---------|
| 1 | 推广充300 | 已执行 | "充了" | 已完成 |
| 2 | 满减改25-3 | 采纳 | "要跟老板报备" | 本周 |
| 3 | 砍月售0的品 | 待定 | "我看看哪些能砍" | 下周 |

## 运营补充的信息
- 满减是上个运营设的，一直没人动
- 厨房只有一个灶，套餐设计要简单
- 老板对推广预算敏感，300是上限

## 未覆盖项
- 无
```

## TODO跟进

TODO到期了，问运营：

```
"上周说满减这周改好，改了吗？"
```

- 运营说改了 → 标完成，设review_due（7天后）
- 运营说还没 → 记录，3天后再问
- 所有TODO完成 → state.step = "executing"

## Review

review_due到了：

```
"满减改了一周了，我帮你看下数据。"
```

用 backend-diagnosis 的能力登后台拉数据，或者问运营要数据。对比执行前后：

```markdown
# Review：店铺名
日期：2026-04-27

## 执行的动作
满减从38-1改为25-3（2026-04-20执行）

## 数据对比（7天）
| 指标 | 改前(7日均) | 改后(7日均) | 变化 |
|------|-----------|-----------|------|
| 日均订单 | 11.9 | 14.2 | +19.3% |
| 下单率 | 25.8% | 29.1% | +3.3pp |
| 客单价 | ¥30.86 | ¥28.50 | -7.6% |

## 结论
部分有效——订单量上去了但客单微降，整体营收持平偏好。
```

state.step = "review_done"

## 超时升级

| 情况 | 动作 |
|------|------|
| 推了诊断2天没回 | 再说一次："XX店的诊断你看了吗？有空聊两句" |
| 催了还是3天没回 | 记录，等Seth查进度时汇报 |
| TODO逾期3天 | 问运营："XX本来说这周做完，卡在哪了？" |
| TODO逾期7天 | 记录升级 |

**不要连续催。催一次等3天，再催一次就停，等Seth处理。**

## 红线

1. **不替运营做决定。** 你是跟进的，不是下指令的。
2. **不改诊断内容。** 运营不认同的记下来，不是你去改判断。
3. **不跳步骤。** 先聊判断再说建议，先确认TODO再追Review。
4. **编号是内部的。** 运营看到的永远是人话，不是P1/A1。
5. **一次不超过5个话题。** 运营没时间，分多次聊。
